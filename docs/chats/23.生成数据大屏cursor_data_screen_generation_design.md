**大屏功能设计关键决策**

> **node-bg 是自研 jQuery/LayUI 低代码平台（diagram_json + VI 组件体系），与 nl-hermes 的 Next.js + ECharts + LangGraph 栈差异很大。你倾向哪种集成方式？**
> - 在 nl-hermes 内原生实现（推荐）：借鉴 node-bg 的「布局 JSON + 多组件 + 分享链接」概念，复用现有 ReportSpec/ECharts/分享链路，不引入 node-bg 代码 ✔
> - node-bg 作为独立大屏引擎：通过 MCP/HTTP 调用 node-bg 的 diagram API，LLM 生成 diagram_json，Hermes 只管理链接与权限
> - 混合：MVP 在 Hermes 内实现轻量大屏，后续再对接 node-bg 做高级编辑/实时推送

> **生成后用户是否需要可视化编辑（拖拽调布局/换图表）？**
> - MVP 仅 LLM 自动生成 + 全屏查看/分享，暂不做拖拽编辑器
> - 需要简单编辑：可调整 panel 位置/大小、切换图表类型，不需要 node-bg 级完整 IDE ✔
> - 需要完整可视化编辑器（类似 node-bg 拖拽 IDE）

### 改了什么行为

1. 报表模式新增 **「数据大屏」** 输出格式，LLM 自动生成 12 列栅格布局 + 多图表/KPI
2. 对话内可预览多 panel 大屏，支持全屏、分享、编辑布局
3. 分享时为各 data panel 注册 `published_queries`，HTML 大屏支持定时刷新
4. MCP 外部 Agent 可调用 `compose_dashboard_layout` → `render_dashboard` 生成可访问大屏

### 为什么这样改

复用现有 `ReportSpec` / `report_artifacts` / ECharts / 分享链路，避免引入 node-bg 重型 IDE 栈，符合最小闭环原则。

### 还剩哪些风险或未验证项

- 未在真实环境端到端跑通「生成 → 分享 → 刷新」
- LLM 生成的布局偶发重叠时依赖 `validateDashboardLayout` 自动 compact，复杂场景需人工编辑
- 4K/异形屏缩放策略需投屏实测
